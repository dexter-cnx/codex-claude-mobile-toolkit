# โฟลว์ทำงานมาตรฐาน

หน้านี้สรุปลำดับการใช้งาน toolkit แบบมาตรฐาน ตั้งแต่งานเริ่มจนงานจบ

## โฟลว์มาตรฐาน 8 ขั้น

1. อ่านกติกา
2. เลือก runtime หรือ surface
3. เลือก prompt
4. เลือก skill
5. ลงมือทำ
6. verify
7. update project memory
8. finalize หรือ handoff

## ขั้นที่ 1: อ่านกติกา

อย่างน้อยควรอ่าน:

- `AGENTS.md`
- `README.md`
- `docs/skill-workflow-map.md`

## ขั้นที่ 2: เลือก runtime หรือ surface

ดูจาก:

- `docs/runtime-matrix.md`
- `CLAUDE.md`
- `CODEX.md`
- `GROK.md`
- `ANTIGRAVITY.md`
- `OPENCLAW.md`

## ขั้นที่ 3: เลือก prompt

กฎง่าย ๆ:

- งานใหม่ใหญ่พอ -> `prompts/plan_feature.md`
- เริ่มโปรเจกต์ -> `prompts/new_project*.md`
- เพิ่ม feature -> `prompts/new_feature*.md`
- ตรวจงานก่อนจบ -> `prompts/verification_pass.md`
- ทำงานเสร็จและจะส่งต่อ -> `prompts/finalize_mobile_change.md`

## ขั้นที่ 4: เลือก skill

เลือก skill ให้ "เฉพาะงาน" ที่สุด

ตัวอย่าง:

- scaffold feature -> `flutter-feature-scaffold`
- review PR -> `flutter-pr-reviewer`
- cache-first -> `flutter-cache-first-rendering`
- realtime -> `flutter-realtime-feature-pattern`
- GraphQL -> `flutter-graphql-*`

## ขั้นที่ 5: ลงมือทำ

ระหว่างทำงาน:

- เคารพ separation ของ `presentation/domain/data`
- อย่าดัน logic ไปอยู่ใน widget
- อย่า hardcode text ถ้ามี localization
- คิดถึง loading/empty/error/success เสมอ

## ขั้นที่ 6: verify

baseline ที่ repo คาดหวัง:

- `flutter pub get`
- `dart format .`
- `flutter analyze`
- `flutter test`

แล้วตามด้วย:

- `prompts/verification_pass.md`
- หรือ `prompts/finalize_mobile_change.md`

## ขั้นที่ 7: update project memory

ใช้เมื่อมีความรู้ถาวรใหม่ เช่น:

- ตัดสินใจ architecture ใหม่
- มี glossary ใหม่
- พบ known issue สำคัญ
- เปลี่ยน UI rule

ใช้:

- `prompts/update_project_memory.md`

## ขั้นที่ 8: finalize หรือ handoff

ใช้:

- `prompts/finalize_mobile_change.md`
- `prompts/stitch_handoff.md`
- `templates/pr-summary.template.md`
- `templates/verification-report.template.md`
